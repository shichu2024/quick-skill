# 提案

## 问题

1. **存量技能质量不可知**：用户拥有大量历史遗留的 AI Skill，无法快速判断哪些符合标准化规范、哪些存在结构缺失，缺乏系统性的合规视图。
2. **改造缺乏精准指引**：即使发现存量技能不合规，用户也不清楚具体哪些项需要修复、哪些可以自动修复、哪些必须人工介入，导致改造工作难以启动。
3. **批量评估成本高**：当存量技能数量较多时，逐一人工检查耗时巨大，缺乏批量扫描和汇总报告能力。
4. **修复标准不统一**：不同维护者对合规标准的理解存在偏差，缺乏统一的诊断维度和量化评分体系。

## 目标

提供一个可对存量技能执行全维度合规性扫描的 CLI 诊断工具，输出量化评分、可视化报告和精准改造清单，并支持对可标准化问题的一键自动修复，为后续 skill-transform 改造提供结构化输入。

## 范围内

- 单技能诊断：对指定路径的存量技能执行 7 个维度的合规性扫描
- 批量诊断：对指定目录下所有存量技能执行批量扫描，输出汇总报告
- 7 个诊断维度：结构合规性、元数据合规性、边界合规性、标准合规性、格式合规性、评测合规性、兼容性合规性
- 每个维度输出修复等级（必须修复 / 建议修复）
- 量化合规评分（0-100 分）
- 可视化诊断报告输出（Markdown 格式）
- 精准改造清单（自动可修复 / 需人工确认分类）
- 自动化修复：对可标准化的诊断项执行一键自动修复（`--fix-auto`）
- 诊断报告输出路径自定义（`--output`）
- 批量诊断按文件类型筛选（`--filter`）

## 范围外

- 不执行技能内容改造（改造由 FEAT-002 skill-transform 负责）
- 不生成测试用例（由 FEAT-004 eval-gen 负责）
- 不执行评测运行（由 FEAT-006 eval 负责）
- 不修改技能核心业务逻辑
- 不涉及 CI/CD 集成
- 不提供 GUI 界面，仅 CLI 输出

## 风险

| 风险类型 | 描述 | 缓解措施 |
|----------|------|----------|
| 诊断标准覆盖度 | 首次定义的 7 个维度可能无法覆盖所有存量技能的特殊形态 | 提供兜底"未知结构"分类，支持迭代扩展诊断规则 |
| 自动修复安全性 | 自动修复可能误改用户的原始文件内容 | 修复前自动备份原始文件，提供 rollback 机制 |
| 大规模批量性能 | 批量诊断数百个技能时可能出现性能瓶颈 | 支持并发数控制，提供进度反馈 |
| 存量技能形态多样性 | 部分存量技能的结构可能完全不符合任何已知模式 | 标记为"无法识别"，输出人工处理建议 |

## 待确认问题

1. 诊断评分的权重分配方案是否需要在 V1 阶段支持用户自定义，还是采用内置默认权重？
2. HTML 格式报告是否纳入 V1，还是仅提供 Markdown 格式？
3. `--fix-auto` 的自动修复范围是否仅限于格式和命名类问题，还是也包含结构目录的自动重组？
4. 兼容性合规性维度的具体检测标准是什么？如何判断"强绑定单一 Agent"？
