# 提案

## 问题

Skill 在生命周期中会持续迭代，SKILL.md 的触发边界、成功标准、执行步骤等内容会频繁修改。当前存在两个核心痛点：

1. **Skill 修改后用例不同步**：开发者修改了 SKILL.md 中的触发场景或完成标准，但测试用例仍然基于旧版本，评测结果失去意义
2. **用户约束无法自动落地到用例**：开发者新增了一条业务约束（如"禁止使用 npm"），但需要手动找到受影响的用例并逐一修改，成本高且容易遗漏

## 目标

实现 SKILL.md 定义与测试用例的全生命周期联动，提供两种核心能力：

1. **技能修改时的用例自动同步**：基于 SKILL.md 与快照版本的差异对比，精准识别变更字段，按映射规则增量调整相关用例
2. **用户约束驱动的用例自动修改**：将用户输入的约束自动分类、写入 SKILL.md 对应章节，同时新增或修改对应测试用例

全程遵循"增量优先、自定义用例保护、冲突可感知"三大原则。

## 范围内

- 变更检测：对比 SKILL.md 与 .skill-snapshot.json 快照，识别变更字段与变更内容
- 影响映射：将字段级变更映射到受影响的用例类型和处理动作（新增/修改/停用）
- 自定义用例保护：custom=true 的用例默认不被自动修改，冲突时仅提示不强制
- 增量同步命令：`quick-skill eval-sync [skill-name]`
- 全量覆盖命令：`quick-skill eval-sync [skill-name] --override`
- 约束驱动命令：`quick-skill eval-sync [skill-name] --constraint "约束内容"`
- 批量同步命令：`quick-skill eval-sync --all`
- 约束自动分类：将用户约束解析为正向触发、负向禁止、成功标准、执行流程、风格规范 5 类
- 用例停用机制：场景被删除时，对应用例标记 deprecated=true 而非物理删除
- 备份保护：同步前自动备份当前用例文件
- 冲突检测：系统生成用例被用户手动修改时，提供三种冲突处理选项

## 范围外

- 约束内容的语义推理（仅做基于关键词和结构的分类匹配）
- 用例的执行与评测（属于 FEAT-006 eval 的职责）
- SKILL.md 内容的人工编辑交互（属于 FEAT-003 skill-create 的职责）
- 跨 Skill 的约束继承或共享用例
- 约束冲突的自动解决（如两条约束互相矛盾）

## 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 变更检测粒度不足，导致用例误修改 | 测试覆盖率下降 | 以 SKILL.md 章节为最小变更单位，不做段落内的 diff 细分 |
| 约束分类错误，导致约束写入错误的 SKILL.md 章节 | Skill 定义偏差 | 对无法明确分类的约束提示用户补充信息，不猜测 |
| 冲突检测遗漏，未发现用户已手动修改的系统用例 | 用户修改被覆盖 | 用例文件增加内容哈希校验，对比生成时与当前的哈希值 |
| --override 全量覆盖导致大量自定义用例丢失 | 数据丢失 | 覆盖前强制二次确认，并自动备份到 .backup/ 目录 |

## 待确认问题

1. **变更检测的最小单位？** 当前假设以 SKILL.md 的章节（When to use this、When NOT to use this 等）为最小变更单位，不检测章节内部的具体段落变化。是否需要更细粒度的 diff？
2. **冲突检测的实现方式？** 当前假设通过对比用例内容哈希来判断系统用例是否被用户修改过。是否需要更明确的冲突标记机制（如在用例中增加 `modified_by` 字段）？
3. **--all 批量同步是否需要并行化？** 建议 MVP 阶段串行执行，后续根据性能数据决定是否引入并发。
4. **约束分类是否需要模型辅助？** 当前假设采用关键词 + 结构规则的确定性分类。是否允许在分类模糊时调用模型辅助判断？
