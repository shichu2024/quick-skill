## ST-001 SKILL.md 变更检测与影响映射

作为 Skill 开发者，我希望 eval-sync 能自动检测 SKILL.md 的变更并识别受影响的用例范围，以便精准同步而不会误改无关用例。

**验收标准**

- AC-001-1：对比当前 SKILL.md 与 .skill-snapshot.json 中存储的快照版本，识别存在差异的章节
- AC-001-2：当 name 或 description 发生变更时，标记所有显式调用用例为"需更新"
- AC-001-3：当 "When to use this" 新增内容时，标记需要新增隐式调用用例和上下文调用用例
- AC-001-4：当 "When to use this" 删除内容时，标记对应场景的用例为"需停用"
- AC-001-5：当 "When NOT to use this" 新增内容时，标记需要新增负例控制用例
- AC-001-6：当 "Definition of done" 发生变更时，标记所有正向用例的 pass_criteria 为"需更新"
- AC-001-7：当 "What to build" 或 "Steps" 发生变更时，标记相关用例的 pass_criteria 和 prompt 为"需检查"
- AC-001-8：当 SKILL.md 与快照无差异时，终止流程并在终端提示"无变更，无需同步"
- AC-001-9：变更检测结果以结构化列表输出，包含章节名称、变更类型（新增/修改/删除）、影响用例类型

**范围外**

- 不检测章节内部的段落级或行级 diff
- 不对变更内容做语义推理

## ST-002 增量用例同步执行

作为 Skill 开发者，我希望通过 `quick-skill eval-sync [skill-name]` 命令自动增量同步用例，以便在修改 SKILL.md 后测试用例能保持一致。

**验收标准**

- AC-002-1：执行命令后，自动完成变更检测、影响映射、用例调整、快照更新的完整流程
- AC-002-2：对于需新增的用例，分配唯一 id，归属正确的用例类型，custom=false
- AC-002-3：对于需修改的用例，更新 prompt 或 pass_criteria 中与变更相关的部分，保留原有 id
- AC-002-4：新增和修改的用例写入原 CSV 文件，保持 CSV 格式合法性
- AC-002-5：同步完成后，更新 .skill-snapshot.json 为当前 SKILL.md 的内容和哈希
- AC-002-6：终端输出同步结果汇总：新增 X 条、修改 X 条、停用 X 条、冲突 X 条
- AC-002-7：同步前自动备份当前用例文件到 ./evals/.backup/ 目录

**范围外**

- 不删除或替换 custom=true 的用例
- 不自动执行同步后的评测

## ST-003 自定义用例保护与冲突处理

作为 Skill 开发者，我希望我手动新增或修改的自定义用例在自动同步过程中被完整保护，不会因系统自动操作而丢失或覆盖。

**验收标准**

- AC-003-1：所有 custom=true 的用例在增量同步中默认被跳过，不做任何修改
- AC-003-2：当检测到系统生成的用例（custom=false）被用户手动修改过（内容哈希与生成时不一致）时，标记为冲突用例
- AC-003-3：冲突用例在同步时暂停处理，在终端展示冲突详情（用例 id、原内容、变更内容）
- AC-003-4：为每条冲突用例提供三种交互选项：保留用户版本、覆盖为系统版本、手动合并
- AC-003-5：用户选择"保留用户版本"时，跳过该用例的修改，并将该用例标记为 custom=true
- AC-003-6：用户选择"覆盖为系统版本"时，执行系统自动修改
- AC-003-7：无冲突时自动完成同步，不弹出交互提示

**范围外**

- 不提供自动合并功能（"手动合并"选项意味着用户在编辑器中自行修改）
- 不检测 custom=true 用例与 SKILL.md 变更之间的潜在不一致

## ST-004 用例停用与快照更新

作为 Skill 开发者，我希望被删除场景对应的用例通过停用标记保留在用例集中，而不是被物理删除，以便保留历史记录和后续可能的恢复。

**验收标准**

- AC-004-1：当 "When to use this" 中某场景被删除时，对应的隐式/上下文调用用例标记为 deprecated=true
- AC-004-2：当 "When NOT to use this" 中某场景被删除时，对应的负例用例标记为 deprecated=true
- AC-004-3：停用操作不物理删除用例行，保留在 CSV 文件中
- AC-004-4：同步完成后，.skill-snapshot.json 更新为当前 SKILL.md 的完整内容和新的版本哈希
- AC-004-5：终端明确列出本次停用的用例 id 和对应的被删除场景

**范围外**

- 不提供自动恢复停用用例的功能
- 不检测停用用例与其他用例之间的依赖关系

## ST-005 用户约束解析与分类

作为 Skill 开发者，我希望输入一条约束后，系统能自动将其分类到正确的 SKILL.md 章节并识别对应的用例处理逻辑，以便约束能准确落地。

**验收标准**

- AC-005-1：能识别正向触发约束（如"当用户需要 X 时可使用"），分类到 "When to use this"
- AC-005-2：能识别负向禁止约束（如"禁止在 Y 场景下使用"），分类到 "When NOT to use this"
- AC-005-3：能识别成功标准约束（如"必须生成符合 Z 规范的产出"），分类到 "Definition of done"
- AC-005-4：能识别执行流程约束（如"必须使用 pnpm，禁止使用 npm"），分类到 "What to build" 或 "Steps"
- AC-005-5：能识别风格规范约束（如"所有组件必须使用函数式写法"），分类到 "What to build"
- AC-005-6：当约束内容无法明确分类时，不猜测分类，而是在终端提示用户补充明确信息
- AC-005-7：一条约束可能同时匹配多个分类，输出所有匹配的分类供用户确认

**范围外**

- 不对约束内容做语义推理或扩展
- 不检测多条约束之间的矛盾或冲突

## ST-006 约束驱动的 SKILL.md 更新与用例调整

作为 Skill 开发者，我希望通过 `quick-skill eval-sync [skill-name] --constraint "约束内容"` 命令一次性完成约束落地到 SKILL.md 和对应用例的调整，以便"有约束必有测试"。

**验收标准**

- AC-006-1：执行命令后，自动完成约束解析、SKILL.md 更新、用例调整、快照更新的完整流程
- AC-006-2：约束内容自动追加到 SKILL.md 对应章节的末尾，不覆盖已有内容
- AC-006-3：SKILL.md 更新前，自动备份上一版本到 .backup/ 目录
- AC-006-4：正向触发约束落地后，自动新增至少 1 条隐式调用用例
- AC-006-5：负向禁止约束落地后，自动新增至少 1 条负例控制用例
- AC-006-6：成功标准约束落地后，更新所有正向用例的 pass_criteria，新增对应校验规则
- AC-006-7：执行流程约束和风格规范约束落地后，更新相关用例的 pass_criteria，新增对应校验用例
- AC-006-8：终端输出约束落地结果：写入的 SKILL.md 章节、新增/修改的用例数量
- AC-006-9：提示用户可执行 `quick-skill eval` 验证约束效果

**范围外**

- 不自动执行评测验证
- 不处理约束之间的潜在冲突

## ST-007 全量覆盖同步与备份保护

作为 Skill 开发者，我希望在需要彻底重建用例集时可以通过 --override 参数全量重新生成，同时系统确保旧数据有备份可恢复。

**验收标准**

- AC-007-1：执行 `quick-skill eval-sync [skill-name] --override` 时，全量重新生成该 Skill 的用例
- AC-007-2：覆盖前自动备份当前用例文件到 ./evals/.backup/ 目录，备份文件名包含时间戳
- AC-007-3：覆盖前在终端提示即将覆盖的用例文件路径，要求用户输入确认（y/N）
- AC-007-4：用户拒绝确认时，终止覆盖流程，不修改任何文件
- AC-007-5：覆盖后更新 .skill-snapshot.json 为当前 SKILL.md 内容和哈希
- AC-007-6：覆盖后的用例集中，原有的 custom=true 用例被重新导入（从备份中恢复），不丢失

**范围外**

- 不提供覆盖后的自动回滚命令

## ST-008 批量同步所有 Skill

作为 Skill 维护者，我希望通过 `quick-skill eval-sync --all` 命令一次性检测并同步所有 Skill 的用例，以便在批量修改 SKILL.md 后快速完成对齐。

**验收标准**

- AC-008-1：自动扫描 ./skills/ 下所有业务分类子目录中的 Skill
- AC-008-2：为每个存在 SKILL.md 变更的 Skill 执行增量同步流程
- AC-008-3：跳过无变更的 Skill，并在终端明确列出跳过的 Skill 名称
- AC-008-4：执行完成后输出汇总信息：已同步数量、跳过数量、失败数量及失败原因
- AC-008-5：单个 Skill 同步失败不阻塞其他 Skill 的同步流程
- AC-008-6：全部同步完成后返回退出码 0（全部成功）或 1（存在失败）

**范围外**

- 不支持批量同步时的并发控制参数
- 不支持按业务分类筛选批量同步范围
