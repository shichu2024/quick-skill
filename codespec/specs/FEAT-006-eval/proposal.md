# 提案

## 问题

Skill 开发者在迭代 Skill 的过程中，缺乏可靠的手段来量化评估 Skill 质量、检测行为回归、定位执行问题。当前痛点：

1. **效果不可衡量**：Skill 是否按预期触发、执行结果是否符合标准，缺少客观量化评估
2. **回归无法感知**：Skill 迭代后，无法自动检测已有用例是否出现新的失败或分数下降
3. **优化无数据支撑**：缺少执行链路追踪和效率指标，难以定位 Skill 的性能瓶颈

## 目标

构建面向 quick-skill 生态的标准化 Skill 评测引擎，提供双引擎评测体系（确定性检查 + 模型辅助 Rubric）、多维度量化打分、执行链路追踪、结果持久化与回归检测能力。仅测试 ./skills/ 下的原始 Skill 源，解决 Skill 迭代中"效果不可衡量、回归无法感知、优化无数据支撑"的核心痛点。

## 范围内

**评测执行**
- 单 Skill 评测：`quick-skill eval [skill-name]`
- 批量评测：`quick-skill eval [skill1,skill2,...]`
- 全量评测：`quick-skill eval --all`

**双引擎评测体系**
- 确定性评测引擎（必选）：结果目标检查、流程目标检查、风格目标检查、效率目标检查
- 模型辅助 Rubric 定性评测引擎（可选）：JSON Schema 评分模板，固定字段 overall_pass/score/checks

**多维度打分**
- 单条用例：确定性检查通过率 + Rubric 加权，满分 100
- 单个 Skill：正例通过率 50% + 负例准确率 30% + Rubric 平均分 20%，满分 100
- 全量 Skill：平均分、Top/末位排行、回归风险、健康度

**执行链路追踪**
- JSONL 格式 trace 日志，记录触发、命令执行、文件操作、token 用量、模型调用事件
- 失败时自动输出 trace 路径，支持一键复现

**结果持久化**
- 本地 .quick-skill-eval/ 目录
- JSON 结果 + trace 日志 + HTML 报告
- 按时间+分类+名称三级归档
- 回归检测：对比上次结果，标注新增失败和得分下降

**测试环境**
- 每条用例独立沙箱
- 默认只读模式
- 单条超时可配置（最长 30 秒）
- 批量并发数可配置（默认 5）

## 范围外

- CI/CD 流水线集成（不包含 GitHub Actions 或其他 CI 系统的适配）
- Agent 目录下平铺部署后的 Skill 测试（仅测试 ./skills/ 下的原始 Skill 源）
- 用例生成和同步（属于 FEAT-004/FEAT-005 的职责）
- 测试产物的自动修复（不根据失败结果自动修改 SKILL.md）
- 发布门禁的自动化执行（属于 FEAT-008 publish 的职责，本模块仅提供评测能力）
- PDF 格式的报告导出

## 风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 确定性检查规则覆盖不足，大量场景需要模型辅助评测 | 评测效率低、成本高 | 优先对 4 类检查项定义核心规则，明确规则无法覆盖的场景才走模型辅助 |
| 沙箱隔离不充分导致测试污染原始 Skill 源 | 项目文件损坏 | 沙箱默认只读，写入操作限定在临时目录内，测试完成后自动清理 |
| 模型辅助评测的 API 调用成本不可控 | 用户费用超预期 | 模型辅助为可选功能，默认关闭；执行前展示预估调用次数和费用提示 |
| 批量评测时并发导致系统资源争用 | 测试结果不稳定 | 默认并发数 5，可配置；单条超时 30 秒上限 |
| HTML 报告模板维护成本 | 功能迭代负担 | 使用轻量级模板引擎，避免引入重前端依赖 |

## 待确认问题

1. **确定性检查规则的具体定义？** 4 类检查项（结果/流程/风格/效率）各自需要定义具体的检查规则。当前假设从 SKILL.md 的 "Definition of done"、"Steps"、"What to build" 中自动提取检查规则，是否需要额外的配置文件？
2. **模型辅助评测的模型选型？** Rubric 定性评测需要调用 LLM 进行结构化输出。当前假设使用与 CLI 一致的模型（如 OpenAI GPT 或 Claude），是否需要支持多模型切换？
3. **沙箱实现的底层机制？** 当前假设使用临时目录 + 文件系统隔离。是否需要考虑 Docker 容器级别的隔离？
4. **HTML 报告的详细程度？** 是否需要包含每条用例的完整执行 trace，还是仅包含失败用例的 trace？建议失败用例完整 trace + 通过用例摘要。
5. **回归检测的历史结果保留策略？** 是否只对比上一次结果，还是需要支持任意两次结果的对比？建议 MVP 仅对比上一次。
