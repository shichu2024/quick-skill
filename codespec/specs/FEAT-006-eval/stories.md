## ST-001 用例加载与解析

作为评测引擎，我需要从 Skill 的 evals 目录加载并解析测试用例，以便后续评测流程消费结构化的用例数据。

**验收标准**

- AC-001-1：能读取 `./skills/[分类]/[skill-name]/evals/[skill-name].prompts.csv` 文件
- AC-001-2：正确解析 CSV 格式，处理含逗号、换行、双引号的字段转义
- AC-001-3：每条用例解析为包含 id、should_trigger、prompt、pass_criteria、custom、deprecated 的结构化对象
- AC-001-4：自动跳过 deprecated=true 的用例，不纳入评测执行
- AC-001-5：当 CSV 文件不存在或格式错误时，输出明确的错误提示，不中断其他 Skill 的评测
- AC-001-6：当 CSV 中存在缺少必填字段的行时，跳过该行并在终端提示用例 id 及缺失字段

**范围外**

- 不支持 JSON 格式的用例文件
- 不对用例内容做语义校验

## ST-002 独立沙箱环境搭建

作为评测引擎，我需要为每条用例创建独立的沙箱执行环境，确保测试执行不污染原始 Skill 源和用户项目目录。

**验收标准**

- AC-002-1：每条用例执行前，在系统临时目录下创建独立的临时沙箱目录
- AC-002-2：沙箱目录内复制目标 Skill 的必要文件（SKILL.md 及 Skill 声明的依赖文件）
- AC-002-3：默认以只读模式挂载原始 Skill 源，测试产物的写入限定在沙箱临时目录内
- AC-002-4：用例执行完成后（无论成功或失败），自动清理沙箱临时目录
- AC-002-5：单条用例超时时间可配置，默认 10 秒，最大不超过 30 秒
- AC-002-6：超时后自动终止用例执行进程，标记为失败，不残留后台进程

**范围外**

- 不使用 Docker 容器级隔离
- 不做网络访问限制

## ST-003 确定性评测引擎

作为 Skill 开发者，我希望评测引擎通过轻量、可复现的规则化检查来评估 Skill 的核心行为，以便获得客观、可调试的评测结果。

**验收标准**

- AC-003-1：**结果目标检查** — 验证任务是否完成：核心输出文件/目录是否按预期生成；命令退出码是否符合预期；最终产物是否满足 SKILL.md 中 "Definition of done" 的可量化标准
- AC-003-2：**流程目标检查** — 验证执行过程是否合规：Skill 是否按 should_trigger 预期被触发或不触发；执行步骤是否与 "Steps" 定义一致，无跳步或多余步骤；命令执行顺序是否符合预期
- AC-003-3：**风格目标检查** — 验证产出风格是否符合约定：产物文件结构和命名是否符合 "What to build" 中的约定；代码/配置格式是否符合预设规则；输出内容是否满足格式约束
- AC-003-4：**效率目标检查** — 验证执行效率是否达标：命令执行总次数在预设阈值内（无循环或无效执行）；总 token 用量在预设阈值内；执行时长未超过超时限制
- AC-003-5：每类检查输出明确的 pass/fail 结果和详细说明
- AC-003-6：4 类检查全部通过时，确定性评测项记为满分；每类检查独立计分
- AC-003-7：当 SKILL.md 缺少某类检查所需的章节时（如无 "Steps" 章节），该类检查标记为"不适用"而非失败

**范围外**

- 不在确定性引擎中调用 LLM
- 不处理 SKILL.md 中无法量化的主观标准

## ST-004 模型辅助 Rubric 评测引擎

作为 Skill 开发者，我希望在确定性规则无法覆盖的定性场景中，通过结构化的模型辅助评分获得补充评估，以便全面衡量 Skill 质量。

**验收标准**

- AC-004-1：支持用户自定义 JSON Schema 评分模板，强制输出结构化结果
- AC-004-2：标准 Schema 包含固定核心字段：overall_pass（布尔值）、score（0-100 整数）、checks（数组，每项含 id、pass、notes）
- AC-004-3：支持自定义评分维度、权重和通过阈值
- AC-004-4：模型辅助评测为可选功能，默认不启用，通过命令参数显式开启
- AC-004-5：评分执行采用只读模式，仅读取测试产物，不修改任何文件，不执行任何命令
- AC-004-6：模型调用失败时，内置重试机制（默认 3 次），超过重试次数标记为失败并记录错误详情

**范围外**

- 不内置默认 Rubric 模板（用户需自行定义或从预设模板中选择）
- 不支持模型辅助评测的并发调用优化

## ST-005 多维度量化打分

作为 Skill 开发者，我希望评测引擎输出可量化的分数，以便横向对比不同 Skill 的质量、纵向追踪同一 Skill 的迭代效果。

**验收标准**

- AC-005-1：**单条用例打分**：基于确定性检查通过率和 Rubric 评分加权计算，满分 100 分，明确标注每项扣分原因
- AC-005-2：**单个 Skill 打分**：正例用例通过率权重 50% + 负例用例准确率权重 30% + 定性 Rubric 平均分权重 20%，满分 100 分
- AC-005-3：当未启用模型辅助评测时，单个 Skill 打分公式调整为：正例通过率 60% + 负例准确率 40%
- AC-005-4：**全量 Skill 打分**：输出全量 Skill 的平均分、通过率 Top 3 和末位 3 排行、整体健康度评分
- AC-005-5：所有分数计算过程透明，分数详情可追溯到具体的检查项和用例

**范围外**

- 不支持自定义权重配置（后续迭代可考虑）
- 不计算跨 Skill 的相关性指标

## ST-006 执行链路追踪

作为 Skill 开发者，我希望评测引擎记录完整的执行链路，以便在测试失败时快速定位问题节点并复现执行过程。

**验收标准**

- AC-006-1：所有测试执行输出 JSONL 格式的结构化 trace 日志
- AC-006-2：trace 事件类型至少包含：Skill 触发事件、命令执行事件、文件操作事件、token 用量事件、模型调用事件
- AC-006-3：每条 trace 事件包含时间戳、用例 id、事件类型、事件详情、执行结果
- AC-006-4：trace 事件顺序与实际执行顺序完全一致
- AC-006-5：用例失败时，终端自动输出对应的 trace 日志本地路径
- AC-006-6：失败节点对应的 trace 事件包含失败原因和上下文信息

**范围外**

- 不提供实时 trace 查看功能（仅事后分析）
- 不将 trace 日志发送到远程服务

## ST-007 测试结果持久化与归档

作为 Skill 开发者，我希望评测结果自动持久化到本地，以便后续查阅历史结果、生成报告和进行回归检测。

**验收标准**

- AC-007-1：测试完成后自动将全量测试结果持久化到 `./.quick-skill-eval/` 目录
- AC-007-2：持久化内容包含：JSON 格式的全量测试结果文件、结构化 trace 日志文件、HTML 可视化报告
- AC-007-3：归档规则为「测试时间/Skill 业务分类/Skill 名称」三级目录结构
- AC-007-4：JSON 结果文件包含所有用例的执行情况、打分详情、检查项明细、trace 文件引用路径
- AC-007-5：持久化操作不影响评测流程的性能（异步写入或写入完成后返回）
- AC-007-6：当目标目录无写入权限时，输出明确错误提示并终止

**范围外**

- 不支持远程持久化（如 S3、数据库）
- 不自动清理历史归档文件

## ST-008 回归检测与版本对比

作为 Skill 开发者，我希望评测引擎能自动对比当前结果与历史结果，标注新增的失败和得分下降，以便及时发现回归问题。

**验收标准**

- AC-008-1：评测完成后，自动查找该 Skill 上一次的测试结果文件
- AC-008-2：对比当前结果与上一次结果，标注以下回归项：新增失败的用例 id、得分下降的用例 id 和下降幅度、整体分数变化
- AC-008-3：发现回归项时，在终端输出回归警告，包含回归用例数量和影响范围
- AC-008-4：无历史结果时，标注为"首次评测"，不执行回归对比
- AC-008-5：回归检测结果写入当前测试结果的 JSON 文件中，供报告生成消费

**范围外**

- 不支持任意两次历史结果的对比（仅对比上一次）
- 不支持跨 Skill 的回归关联分析

## ST-009 单 Skill 评测命令

作为 Skill 开发者，我希望通过 `quick-skill eval [skill-name]` 命令测试单个 Skill，以便在迭代后快速验证行为是否符合预期。

**验收标准**

- AC-009-1：执行命令后，自动在 ./skills/ 的所有业务分类子目录中查找匹配的 Skill 目录
- AC-009-2：找到目标 Skill 后，自动加载用例、创建沙箱、执行确定性评测、计算打分、持久化结果
- AC-009-3：若目标 Skill 不存在，输出明确错误提示并退出
- AC-009-4：若目标 Skill 无用例文件，提示用户先执行 eval-gen 并退出
- AC-009-5：评测过程中在终端展示进度条、当前用例 id、实时通过/失败状态
- AC-009-6：评测完成后输出汇总：平均分、用例通过率、回归风险提示
- AC-009-7：命令执行成功返回退出码 0（全部通过）或 1（存在失败）

**范围外**

- 不支持在命令行指定单个用例 id 执行
- 不支持自定义结果输出目录参数

## ST-010 批量与全量评测命令

作为 Skill 维护者，我希望通过批量或全量评测命令一次性测试多个 Skill，以便快速掌握整体质量状况。

**验收标准**

- AC-010-1：执行 `quick-skill eval [skill1,skill2,...]` 批量测试指定的多个 Skill
- AC-010-2：执行 `quick-skill eval --all` 测试 ./skills/ 下全量 Skill
- AC-010-3：批量评测的并发数可配置，默认 5，通过 --concurrency 参数控制
- AC-010-4：单个 Skill 评测失败不阻塞其他 Skill 的评测流程
- AC-010-5：全部评测完成后输出极简汇总：总测试 Skill 数、整体通过率、平均分、Top 3 高风险 Skill、回归项数量
- AC-010-6：支持 --category 参数按 Skill 业务分类筛选测试范围
- AC-010-7：支持 --incremental 参数，仅测试上次评测后有文件变更的 Skill

**范围外**

- 不支持跨项目或跨仓库的批量评测
- 不支持评测结果的远程推送

## ST-011 HTML 可视化报告生成

作为 Skill 开发者，我希望评测完成后自动生成可视化 HTML 报告，以便通过浏览器直观查看评测详情、失败用例和回归提示。

**验收标准**

- AC-011-1：评测完成后自动在归档目录下生成 HTML 格式的可视化报告
- AC-011-2：报告包含以下核心板块：得分概览（Skill 级别和用例级别的分数）、失败用例详情（含 prompt、预期结果、实际结果）、回归提示（与上次对比的变化项）、trace 日志跳转链接
- AC-011-3：失败用例的 trace 日志可在报告中通过链接直接跳转查看
- AC-011-4：报告使用轻量级 HTML 模板，不依赖外部 CSS/JS 资源，可离线打开
- AC-011-5：评测完成后，在终端输出 HTML 报告的本地文件路径

**范围外**

- 不生成交互式的 Web 应用（仅为静态 HTML 报告）
- 不支持报告的自定义主题或样式
- 不支持 PDF 格式导出
