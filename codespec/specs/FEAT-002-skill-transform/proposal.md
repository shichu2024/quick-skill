# 提案

## 问题

1. **存量技能无法纳入标准化管理**：大量非标准化存量技能无法直接被 quick-skill 生命周期管理体系管理，无法部署、评测和发布。
2. **手工改造成本高且质量不一**：将存量技能手工转化为标准化格式需要深入理解 Anthropic Skill 规范，耗时长且容易遗漏关键要素。
3. **改造风险不可控**：改造过程中可能误改核心业务逻辑，缺乏回滚机制和改造前后对比验证。
4. **特殊形态技能无法处理**：强绑定单一 Agent、无明确边界、带复杂脚本等特殊技能缺乏兜底方案，改造流程直接中断。

## 目标

基于 FEAT-001 skill-diagnose 的诊断结果，将非标准化存量技能自动化转化为符合 Anthropic 官方规范和 quick-skill 管理体系的标准化技能，全程最小化人工干预，不修改技能核心业务逻辑，并提供完整的回滚和验证机制。

## 范围内

- 全流程 5 阶段改造：
  1. 结构标准化改造（自动化）：目录归一化、核心文件提取、冗余清理、版本快照备份
  2. SKILL.md 标准化补齐（自动生成 + 人工确认）：元数据、触发边界、核心逻辑、执行步骤、完成标准
  3. 测试用例自动化补齐（自动化）：调用 eval-gen 生成用例、快照生成、自定义用例兼容
  4. 合规性验证与评测（自动化）：二次诊断、触发测试、全量评测、问题自动修复
  5. 纳入生命周期管理：版本锁定、体系纳入、eval-sync 联动配置、多 Agent 适配验证
- 单技能精细化改造
- 批量改造：批量诊断排序、批量结构归一化、批量 SKILL.md 生成、批量用例补齐与评测
- 回滚能力（`--rollback`）：恢复到原始版本
- 改造前自动备份到 `.backup/` 目录
- 兜底兼容方案：
  - 强绑定单一 Agent 的技能
  - 无明确边界的通用技能
  - 带复杂配套脚本的技能
  - 历史遗留无维护的技能
  - 格式特殊的技能
- 批量改造进度可视化、暂停与续跑
- 改造验收准入标准检查

## 范围外

- 不修改技能核心业务逻辑（最小侵入原则）
- 不创建新技能（由 FEAT-003 skill-create 负责）
- 不独立执行评测（依赖 FEAT-006 eval 模块）
- 不独立生成用例（依赖 FEAT-004 eval-gen 模块）
- 不涉及 Skill 发布（由 FEAT-008 publish 负责）
- 不提供 GUI 界面

## 风险

| 风险类型 | 描述 | 缓解措施 |
|----------|------|----------|
| 依赖链复杂 | 依赖 FEAT-001（诊断）、FEAT-004（用例生成）、FEAT-006（评测）三个模块 | 采用松耦合接口调用，任一依赖不可用时降级为仅执行前置阶段 |
| 核心逻辑误修改 | 自动提取和重组过程中可能改变技能的语义 | 严格遵循"最小侵入"原则，核心逻辑原文保留，仅做结构包装 |
| 批量改造中断 | 大规模批量改造中某个技能失败可能影响整体流程 | 单技能失败不阻塞后续技能，支持从断点续跑 |
| 兜底方案覆盖度 | 特殊形态技能的兜底方案可能无法覆盖所有边缘情况 | 提供"标记为归档"降级策略，不进入发布流程 |
| SKILL.md 生成质量 | 自动生成的 SKILL.md 内容准确性依赖对原始技能的解析质量 | 人工确认环节作为必经步骤，支持修改和补充 |

## 待确认问题

1. 改造后技能的版本号起始规则是什么？是否所有改造后的技能都从 1.0.0 开始？
2. SKILL.md 人工确认环节是否支持跳过（即纯自动化改造）？跳过后的准入标准是什么？
3. 批量改造的并发数默认值和上限分别是多少？
4. 兜底方案中"归档状态"的技能是否允许后续被重新激活进入发布流程？
5. 改造验收准入标准中"正向用例通过率 >= 90%"是否为硬性门禁？
