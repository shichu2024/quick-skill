## ST-001 结构标准化改造

**作为** Skill 维护者，**我希望**工具自动将存量技能的目录结构归一化为标准化格式并备份原始版本，**以便**在不丢失原始内容的前提下完成结构层面的合规化。

### 验收标准

- **AC-1**：改造前自动在 `.backup/` 目录生成原始技能的完整快照（包含所有文件和目录结构）。
- **AC-2**：自动创建标准化技能目录结构 `[业务分类]/[skill-name]/`，将存量技能的核心文件迁移到该目录内。
- **AC-3**：自动从存量技能的文本内容中提取核心业务逻辑、执行步骤和核心能力描述。
- **AC-4**：自动识别并清理无关的临时文件、备份文件和冗余日志。
- **AC-5**：迁移完成后，存量技能的所有原始文件内容完整保留在快照中，可随时还原。
- **AC-6**：当存量技能已处于标准化目录结构时，跳过结构迁移步骤，仅执行快照备份。

### 范围外

- 不生成 SKILL.md（由 ST-002 负责）
- 不修改技能核心业务逻辑

## ST-002 SKILL.md 标准化补齐

**作为** Skill 维护者，**我希望**工具基于存量技能内容自动生成标准化的 SKILL.md，并在写入前提供交互式确认，**以便**我能在最小人工干预下获得符合 Anthropic 规范的技能定义文件。

### 验收标准

- **AC-1**：自动生成包含完整 YAML front matter 的 SKILL.md，front matter 包含 `name`（自动转为 kebab-case）和 `description`（精准概括核心能力）。
- **AC-2**：自动生成"When to use this"章节，明确列出适用场景。
- **AC-3**：自动生成"When NOT to use this"章节，明确列出禁止场景。
- **AC-4**：自动生成"What to build"章节，结构化整理技能的核心产出物、规范要求和用户约束。
- **AC-5**：自动生成"Steps"章节，提取存量技能的执行流程为有序步骤。
- **AC-6**：自动生成"Definition of done"章节，将模糊要求转化为可量化、可检查的完成标准。
- **AC-7**：自动生成后启动交互式确认流程，用户可修改、补充、调整每个章节的内容。
- **AC-8**：用户确认后写入 SKILL.md 文件，覆盖写入前自动备份上一版本。

### 范围外

- 不生成测试用例（由 ST-003 负责）
- 不替代用户编写技能核心逻辑，仅做结构化整理

## ST-003 测试用例自动化补齐

**作为** Skill 维护者，**我希望**工具在 SKILL.md 补齐后自动调用 eval-gen 生成配套测试用例，**以便**技能改造完成后即可具备完整的评测覆盖。

### 验收标准

- **AC-1**：基于已补齐的 SKILL.md 内容，自动调用 eval-gen 模块生成 10-20 条覆盖 4 类核心场景的测试用例。
- **AC-2**：生成的用例输出到技能目录下的 `evals/[skill-name].prompts.csv`。
- **AC-3**：自动生成 `.skill-snapshot.json` 快照文件，记录当前 SKILL.md 的版本哈希。
- **AC-4**：存量技能中已存在的零散测试用例被自动识别并导入用例集，标记为 `custom=true`，不被自动覆盖。
- **AC-5**：eval-gen 模块不可用时，跳过用例生成步骤，输出降级提示，不阻塞后续阶段。

### 范围外

- 不独立实现用例生成逻辑（依赖 FEAT-004 eval-gen 模块）
- 不执行用例评测（由 ST-004 负责）

## ST-004 合规性验证与评测

**作为** Skill 维护者，**我希望**改造完成后工具自动执行二次诊断和全量评测，验证改造效果达标，**以便**确保改造后的技能可以安全纳入管理体系。

### 验收标准

- **AC-1**：改造完成后自动执行 skill-diagnose 二次诊断，验证所有"必须修复"项已完成。
- **AC-2**：二次诊断合规评分达到准入标准（必须修复项全部通过）方可继续，否则输出不达标项清单。
- **AC-3**：自动执行显式调用用例测试，验证技能可被 Agent 稳定触发，无基础故障。
- **AC-4**：调用 eval 模块执行全量测试用例，输出评测报告。
- **AC-5**：评测不通过的项，自动回溯调整 SKILL.md 对应内容，重新生成用例，再次评测，最多重试 3 次。
- **AC-6**：eval 模块不可用时，跳过评测步骤，输出降级提示，标记为"待评测验证"状态。

### 范围外

- 不独立实现评测逻辑（依赖 FEAT-006 eval 模块）
- 不发布技能（由 FEAT-008 publish 负责）

## ST-005 纳入生命周期管理

**作为** Skill 仓库维护者，**我希望**改造完成的技能自动纳入 quick-skill 生命周期管理体系，**以便**该技能后续可被 init 部署、publish 发布、eval-sync 同步。

### 验收标准

- **AC-1**：为改造后的技能生成正式版本号，记录改造前后的版本变更日志。
- **AC-2**：自动将技能同步到 quick-skill 的 `./skills/` 分类目录下。
- **AC-3**：自动配置 eval-sync 联动规则，后续技能修改时可自动同步测试用例。
- **AC-4**：验证技能可在 Claude / OpenCode / Relay 三个 Agent 目录结构中正常部署，无兼容性问题。
- **AC-5**：输出纳入管理的确认信息，包含技能名称、版本号、分类路径。

### 范围外

- 不执行实际的 Agent 部署（由 FEAT-007 init 负责）
- 不发布到 npm 仓库（由 FEAT-008 publish 负责）

## ST-006 改造回滚能力

**作为** Skill 维护者，**我希望**在改造结果不满意时可以一键回滚到改造前的原始状态，**以便**改造过程无风险，随时可以撤销。

### 验收标准

- **AC-1**：通过 `--rollback` 参数指定技能路径，工具将该技能恢复到改造前的原始版本。
- **AC-2**：回滚操作从 `.backup/` 目录的快照中完整还原所有原始文件和目录结构。
- **AC-3**：回滚前确认 `.backup/` 目录存在且快照完整，快照缺失时输出错误提示并终止回滚。
- **AC-4**：回滚完成后输出回滚确认信息，包含：技能名称、回滚前版本、回滚后版本、备份路径。
- **AC-5**：回滚操作删除改造过程中生成的所有文件（SKILL.md、evals/ 目录等），仅保留原始内容。

### 范围外

- 不支持选择性回滚（仅支持完整回滚到改造前状态）
- 不删除 `.backup/` 目录本身

## ST-007 批量改造与进度可视化

**作为** Skill 仓库维护者，**我希望**对目录下所有存量技能执行批量改造，并实时查看改造进度，**以便**高效完成大规模技能标准化工作。

### 验收标准

- **AC-1**：给定一个目录路径，工具自动扫描所有子目录，按诊断评分从低到高排序后依次执行改造。
- **AC-2**：批量改造过程展示实时进度：当前技能 / 总数、当前阶段（结构改造 / SKILL.md 补齐 / 用例补齐 / 验证 / 纳入管理）。
- **AC-3**：单个技能改造失败不阻塞其他技能的改造，失败项记录错误原因并继续处理下一个。
- **AC-4**：批量改造完成后输出汇总报告：总改造数、成功数、失败数、各技能改造前后合规评分对比。
- **AC-5**：支持通过 `--concurrency` 参数控制批量改造的并发数。
- **AC-6**：支持中断后续跑，跳过已成功改造的技能，从第一个未完成的技能继续。

### 范围外

- 不支持 GUI 进度界面
- 不支持远程批量改造（仅限本地目录）

## ST-008 特殊形态技能兜底兼容

**作为** Skill 维护者，**我希望**工具能妥善处理强绑定 Agent、无明确边界、带复杂脚本等特殊形态的存量技能，**以便**这些技能不会因改造流程而中断或损坏。

### 验收标准

- **AC-1**：对强绑定单一 Agent 的技能，保留原有核心逻辑，在 SKILL.md 中明确标注适配的 Agent 类型，跳过跨 Agent 兼容性验证。
- **AC-2**：对无明确业务边界的通用技能，自动生成最小集元数据和完成标准，标注为"通用技能"分类，保留原有 prompt 内容。
- **AC-3**：对带复杂配套脚本的技能，保留原有脚本目录结构，在 SKILL.md 中明确脚本执行要求和权限要求。
- **AC-4**：对历史遗留无维护的技能，自动生成基础结构与元数据，标记为"归档状态"，不进入发布流程。
- **AC-5**：对格式特殊的技能，支持自定义目录结构映射规则，仅补齐元数据和用例，不强制修改原有结构。
- **AC-6**：每个兜底方案执行后，在改造报告中标注兜底类型和处理方式，与标准改造结果明确区分。

### 范围外

- 不保证兜底方案处理后的技能能通过所有标准准入验证
- 不自动判断技能属于哪种特殊形态（需要用户指定或确认）
